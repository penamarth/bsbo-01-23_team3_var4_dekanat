## Рамки
Приложение информационной системы деканата образовательного учреждения.

## Уровень  
Задача, определенная пользователем.

## Основной исполнитель  
Сотрудник деканата.

## Заинтересованные лица и их требования  
- Сотрудник деканата: Хочет правильно и быстро оформить отчисление, минимизировать рутинные операции, избежать ошибок в документации
- Декан: Хочет контролировать процесс отчисления, получать уведомления о критичных случаях
- Студент: Хочет получить официальное подтверждение отчисления и справку об обучении
- Учебно-методический отдел: Хочет своевременно получать данные об отчислении для корректировки учебных групп и отчетности
- Бухгалтерия: Хочет получать информацию об отчислении для прекращения начислений стипендий
- Государственные образовательные службы: Требуют соблюдения законодательства при отчислении и своевременной передачи данных.

## Предусловия  
1. Сотрудник деканата аутентифицирован в информационной системе.  
2. Студент числится в базе данных информационной системы.  
3. Имеются основания для отчисления (академическая задолженность, личное заявление, выговоры и т.д.).

## Результаты (Постусловия)
- Статус студента изменен на "отчислен" в StudentsService
- Студент удален из группы через GroupsService
- Академические записи студента архивированы в AcademicRecordsService
- Документы об отчислении зарегистрированы в DigitalDocumentManager
- Учебно-методический отдел уведомлен через EducationalDepartmentService
- Бухгалтерия уведомлена о прекращении выплат

## Основной успешный сценарий
1. Сотрудник деканата выбирает операцию "Отчисление студентов" в UI
2. Система через DeanOfficeController вызывает getStudentInfo() из IStudentsService
3. Сотрудник вводит идентификатор студента (номер зачётной книижки или ФИО)
4. Система отображает профиль студента и его текущую группу
5. Сотрудник выбирает основание для отчисления из списка
6. Система проверяет академические записи студента через IAcademicRecordsService
7. Сотрудник подтверждает отчисление
8. Система выполняет:
   - expelStudent() через IStudentsService
   - removeStudent() из группы через IGroupsService
   - registerDocument() приказа об отчислении через IDigitalDocumentManager
9. DigitalDocumentManager отправляет документ на подписание декану через sendForSigning()
10. После подписания система:
    - sendTransferRequest() в учебно-методический отдел через IEducationalDepartmentService
    - checkScholarship() прекращение выплат через Accounting
11. Система формирует уведомление для студента
12. Процесс отчисления завершен
    
## Расширения (альтернативные потоки)
**а. Декан отклоняет отчисление**
1. DigitalDocumentManager регистрирует отказ
2. Система восстанавливает студента в исходное состояние
3. Сотрудник деканата уведомляется об отклонении

**б. Отсутствуют необходимые документы**
1. Система проверяет обязательные документы через DigitalDocumentManager
2. При отсутствии документов процесс приостанавливается
3. Система уведомляет о необходимости предоставить документы

**в. Студент имеет академическую задолженность**
1. Система проверяет academicRecords через AcademicRecordsService
2. При наличии задолженности автоматически выбирается соответсвующее основание
3. Процесс продолжается с дополнительными проверками успеваемости

**г. Ошибка в данных студента**
1. Система обнаруживает несоответствие данных при вызове getStudentInfo()
2. Процесс приостанавливается
3. Требуется корректировка данных через StudentsService

## Специальные требования
- Интерфейс системы должен быть интуитивно понятным и доступным для пользователей с разным уровнем компьютерной грамотности.
- Система должна обеспечивать быстрое выполнение операций (отклик менее 5 секунд).
- Данные должны быть защищены в соответствии с федеральными закони о защите персональных данных.
- Интеграция с DigitalDocumentManager для электронного документооборота
- Синхронизация данных с группами через GroupsService
- Поддержка транзакционности операций отчисления
- Автоматическая проверка академических записей через AcademicRecordsService

## Список технологий и типов данных
- Использование репозиториев (IStudentsRepository, IGroupsRepository) для доступа к данным
- Работа с DbContext для сохранения изменений
- Электронная подпись документов через DigitalDocumentManager
- Интеграция с внешними системами через определенные интерфейсы

## Частота использования
Несколько раз в семестр, в зависимости от количества отчислений.
