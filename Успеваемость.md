# **Сценарий использования: Учет успеваемости (формирование ведомости)**

## **Рамки**
Приложение информационной системы деканата образовательного учреждения для учета и документирования академической успеваемости студентов.

## **Уровень**
Задача, определенная пользователем.

## **Основной исполнитель**
Сотрудник деканата (ответственный за учебный процесс).

## **Заинтересованные лица и их требования**
* **Сотрудник деканата:** Хочет оперативно и безошибочно сформировать, проверить и закрыть ведомости по дисциплинам, исключить ручной ввод и дублирование данных.
* **Преподаватель:** Хочет своевременно и в удобной форме внести оценки студентов в систему, получить готовый официальный документ (ведомость) для подписи.
* **Декан:** Хочет иметь актуальный и централизованный доступ к сводной информации об успеваемости по группам и дисциплинам для анализа и принятия решений.
* **Бухгалтерия:** Хочет автоматически получать данные о результатах сессии (академические задолженности) для корректировки выплат стипендий.
* **Студент:** Хочет видеть свои текущие оценки в личном кабинете в режиме, близком к реальному времени, после их утверждения.

## **Предусловия**
1. Сотрудник деканата и Преподаватель идентифицированы и аутентифицированы в системе.
2. В системе заведены учебные планы, дисциплины, группы, прикрепленные к ним студенты и преподаватели.
3. Завершен период сдачи академических работ (экзамен, зачет) по конкретной дисциплине и группе.

## **Результаты (Постусловия)**
* Оценки по дисциплине зафиксированы в системе.
* Электронная ведомость сформирована, подписана ЭП преподавателя и утверждена сотрудником деканата.
* Данные об успеваемости обновлены в карточках студентов.
* Информация о задолженностях/стипендиях передана в Бухгалтерию.
* Оценки отображены в личных кабинетах студентов.

## **Основной успешный сценарий**
1. Преподаватель выбирает в системе дисциплину и группу.
2. Преподаватель вносит оценки ("зачет"/"незачет", баллы) для каждого студента из списка группы.
3. Система автоматически сохраняет каждую оценку в базе данных.
4. Система обновляет академические записи студентов в их личных делах.
5. Система публикует итоговые оценки в личных кабинетах студентов.
6. Процесс выставления оценок завершен.

## **Расширения (альтернативные потоки)**
### **а. Преподаватель не внес оценки в установленный срок.**
1. Система автоматически направляет преподавателю и сотруднику деканата уведомление-напоминание.
2. Сотрудник деканата связывается с преподавателем для выяснения причин.

### **б. Обнаружена ошибка в оценке после утверждения ведомости.**
1. Преподаватель или Сотрудник деканата инициирует создание служебной записки на исправление.
2. После согласования с деканом система создает корректирующую ведомость, которая проходит тот же процесс подписания и утверждения.
3. Система вносит изменения в записи студента с пометкой об исправлении.

### **в. Студент отсутствует в списке на экзамен (новый, переведенный).**
1. Преподаватель запрашивает у сотрудника деканата добавление студента в список группы по дисциплине.
2. Сотрудник деканата вносит изменение в учебный состав группы.
3. Система обновляет список в форме для ввода оценок.

## **Специальные требования**
- Интерфейс должен быть интуитивно понятным при высоких нагрузках.
- Система должна обеспечивать быстрое выполнение операций (отклик менее 3 секунд).
- Данные должны быть защищены в соответствии с ФЗ о защите персональных данных.
- Возможность интеграции с внешними системами.

## **Список технологий и типов данных**
- Использование электронной подписи для приказов.
- Поддержка сканированных копий документов (PDF, JPEG).
- Автоматическая генерация уведомлений по email и SMS.
- Массовая обработка данных.

## **Частота использования**
Массово – в период экзаменационных сессий (2-3 раза в год). Единично – в течение семестра для курсовых проектов, зачетов.
